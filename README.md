# Trip Fuel Prediction — MLOps Pipeline (Practice Project)

A practice implementation of an airline trip-fuel prediction pipeline: a model
predicts expected fuel burn per flight and flags flights that deviate
meaningfully from that prediction, as a proactive fuel-efficiency signal.

> **Note on data**: this project uses synthetic flight data generated by
> `data/generate_data.py`, not a real airline's records. It's a portfolio/practice
> build demonstrating the same MLOps pattern (baseline-gated model training,
> containerized serving, backtested savings methodology) used on a real
> production system I built previously with genuine airline data.

## Architecture

1. **Data generation** (`data/generate_data.py`) — synthetic flights across 10
   routes and 5 aircraft types, with a genuine underlying fuel-burn relationship
   (distance, aircraft type, takeoff weight, headwind) plus realistic measurement
   noise.
2. **Training** (`train.py`) — time-based split (train on 2023, test on 2024, to
   mirror real deployment and avoid leakage). Trains an XGBoost regressor and
   gates deployment against a historical-average baseline — the model only
   "passes" if it beats the baseline on held-out data.
3. **Evaluation** (`evaluate_savings.py`) — flags flights where actual fuel
   exceeded the model's prediction by more than 5%, and expresses the excess
   fuel across flagged flights as a percentage of total fuel burned. This is a
   **backtested potential**, not a measured outcome from flights that were
   actually re-planned.
4. **Serving** (`app/main.py`) — FastAPI service exposing `/predict` (single-flight
   lookup) and `/health` (container readiness).
5. **Containerization** (`Dockerfile`) — multi-stage build, deployed and tested
   locally via Docker Desktop.

## Results (from an actual run against the synthetic dataset)

| Metric | Value |
|---|---|
| Baseline (historical average) MAE | 477 kg |
| Model (XGBoost) MAE | 354 kg |
| Improvement over baseline | 25.8% |
| Backtested potential fuel savings | 0.87% of total fuel burned |
| Flights flagged as inefficient (>5% over prediction) | ~850 of 7,513 test flights |

## Running it

```bash
pip install -r requirements.txt
python data/generate_data.py      # regenerate synthetic data (fixed seed - reproducible)
python train.py                    # train + backtest gate
python evaluate_savings.py         # worked savings calculation
```

## Running it with Docker

```bash
docker build -t trip-fuel-api .
docker run -p 8000:8000 trip-fuel-api
```

Then visit `http://localhost:8000/docs` for an interactive Swagger UI to test
the `/predict` endpoint directly in the browser.

## What this demonstrates

- Baseline-gated model deployment (a model must beat a simple lookup before
  being considered a valid replacement)
- Time-based train/test splitting to avoid leakage and mirror real deployment
- A defensible, reproducible methodology for a business-facing "potential
  savings" metric, rather than an unexplained headline number
- Containerized serving with a working health check
- Path handling anchored to script location (`pathlib`), so the same code runs
  unchanged locally, in Docker, and in a cloud environment like AWS ECS
